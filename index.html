SNAPS: the way JS UX should be

I recently encountered a rendering engine that does so much right that the things that are wrong with it are getting accepted as "the ways that should be."

I am attempting to address these things in code and show how easy it is to do things right.

1: Total DOM control

Instead of having widgets which seem like they are ashamed of the DOM, SNAPS embraces DOM wholly by allowing you to fully access and set all parts of a DomElement (attributes, styles, content), not just a subset of "Blessed" attributes and properties

2: Embracing polyglot programming

Instead of using fully customized physics, 3D and event systems, Snaps uses proven, available engines like THREE, Box2D and Signal. This keeps standard systems as part of the frame, and doesn't bloat itself with reinventions of problems that are largely solved.

3: Designed for Threading

Snaps is designed to enable easy export of data across the net, or threads, to take advantage of the emerging goodness of WebWorkers.

4: A broad, open network of shared components

Snaps uses a collection of Snap instances connected with links to form complex patterns that allow intercommunication. Specialized Snap inheritors can manage DOM interactions, or take on simple tasks like managing the size of DOM elements.

Snap instances can be linked in parent child, semantic links, open graphs and sets, to allow any sort of structure you may need; for instance, you can set a row height Snap as the resource link of several different Snap's, and a column width Snap as the resource link of several different Snap's, to enable a grid pattern.

While this pattern was created to enable DOM management, it can be used to manage any number of graph/property/semantic problems.

5: Synchronized updates

As part of the optimization process for distributed work, all of the updating of data is synchronized by a global update() call, allowing a gated output of data to a remote system.
